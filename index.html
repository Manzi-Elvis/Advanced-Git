What I Actually Understand Now

Not command syntax — the reasoning underneath it.

Rebase rewrites. Merge preserves.

A commit's hash depends on its content and its parent's hash. Touch one commit, and everything built on top of it gets a new hash — even if nothing else changed. Rebase leans into that; merge deliberately avoids it.

Nothing is really gone until it's collected.

git reflog tracks every HEAD movement, separately from git log. Almost any mistake — a bad rebase, a wrong reset — is recoverable if I catch it before garbage collection.

Detached HEAD isn't broken — it's just unlabeled.

HEAD normally points at a branch name, which points at a commit. Check out a raw hash, and HEAD points straight at the commit instead. Fine to explore; risky only if I commit and walk away without a branch to hold it.

Conflicts need real divergence.

Two branches only conflict if both sides changed the same lines independently after splitting. Learned this the honest way — my first attempt just fast-forwarded with nothing to resolve.
